# 01-从“布局交付”到“意图交付” (Intent-Driven vs Layout-Driven Handoff)

## 🚧 问题发现：为什么生成的 UI 总是“缺乏设计感”？

在我们在 OpenCode 中早期跑通的工作流里，出现了一个非常经典的模型交互陷阱：

**旧的交付链路：**
1. `用户输入需求`
2. `OpenCode (大语言模型)` 分析需求，并**输出极其详细的文字版原型与交互说明**（例如：顶部是 Banner，下面是 3 个并排的卡片，左边放图右边放字）。
3. `Stitch / Generative UI 工具` 接收这段详细的文本描述，渲染为真实的 UI 界面。

**致命缺陷 (The Bottleneck)：**
* **“越俎代庖的语言模型”：** LLM 本质上是不懂视觉空间构成和美学的。当它试图用文字把“原型布局”定义死的时候，它其实是在做一份非常粗糙且僵化的“线框图”。
* **“沦为填色工具的设计引擎”：** Stitch 等具备专业视觉渲染能力的工具，被 LLM 过度详细的“布局指令”绑架了。它们不敢违背指令，只能像一个“填色工人”一样，在这个僵化的线框上强行贴图和上色。最终导致**画虎不成反类犬**，没有发挥出多模态生成工具在版式、空间、留白和组件重构上的真正实力。

---

## 🎯 重新定义交接物：让设计工具回归“设计”

既然 Stitch 等工具的强项是“视觉表达与组件化构成”，那么 LLM 传递给它的，**不应该是具体的“骨架与布局 (Wireframe)”**，而应该是**“意图、核心数据与体验目标 (Intent, Data & Vibe)”**。

**新的交付链路范式 (Intent-Driven Handoff)：**

### 阶段一：OpenCode (LLM) 负责【收敛意图与提炼数据】
语言模型不再输出具体的界面排版指令，而是输出设计工具真正需要的**核心诉求集合**：
1. **用户核心目标 (User Goal):** 用户在这个页面首要完成的动作是什么？（例如：快速做出购买决策）
2. **必须要暴露的字段 (Data Schema):** 页面必须呈现哪些客观信息？（例如：商品名、价格、销量、5条热评、发货仓）
3. **优先级与信息层级 (Information Hierarchy):** 什么是 P0 级最重要的元素？什么是辅助信息？
4. **情感与氛围 (Vibe / Tone):** 这个页面需要传递什么样的品牌感觉？（例如：生鲜电商需要“新鲜、健康、紧迫感”）

### 阶段二：Stitch / 设计引擎负责【重构视觉与布局呈现】
Stitch 接收到上述“意图集合”后，由其内置的生成引擎自由发挥：
1. **它来决定排版：** 是大图轮播还是瀑布流？是悬浮底部按钮还是侧边栏？交给 Stitch 去随机和探索最佳方案。
2. **它来决定组件：** 是用标签（Tags）还是徽章（Badges），是用卡片（Cards）还是列表（Lists）。
3. **它来决定留白和色彩：** 保证视觉上的呼吸感和现代设计趋势。

---

## 🛠️ 下一步探索：构建“意图协议 (Intent Protocol)”

如果我们认同这个方向，我们需要做的是：
在 OpenCode 和 Stitch 之间，**建立一套“Prompt 协议”**。

这套协议的作用是限制 OpenCode 产生“左边放什么、右边放什么”的废话，强制 OpenCode 只输出“业务诉求和数据结构”，然后把最自由的 Prompt 交给 Stitch 去渲染。
