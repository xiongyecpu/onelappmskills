## 内容规范

### 语言与受众

- **主要语言**：简体中文
- **受众**：产品经理、项目经理、运营负责人、BU高管
- **语气**：专业但易于理解；避免过于学术化的术语
- **领域**：产品管理、市场调研、用户体验、敏捷工作流

### Markdown 风格

- 使用 `#` 作为文档标题（每个文件一个），`###` 作为主要章节，`####` 作为子章节
- 使用 `---` 水平线分隔主要主题块
- 无序列表优先使用 `*`；使用 `●/○/■` 符号表示层级工作流描述（与现有 README.md 风格保持一致）
- 使用粗体（`**文本**`）标注关键术语、强调和重要提示
- 行尾保持自然句子边界（不要在80字符处强制换行）
- 当列表项包含多句话时，在项之间使用空行

### 文件命名

- 内容文件使用描述性的中文名称（如 `市场调研思路.md`）
- 基础设施文件使用英文名称（`README.md`、`AGENTS.md`、`LICENSE`）
- 文件名中不要使用空格——使用下划线或连续的中文字符
- 所有文件均为 `.md`（Markdown）格式

### 文档结构模式

每个方法论文档应遵循以下结构：

```markdown
# 文档标题

### 第N种：方法/场景名称

**核心目的：** 一句话说明目标
**适用场景：** 何时使用此方法

#### 1. 分析维度名称（简短括号备注）
* **子项名称：** 详细说明
* **调研方法：** 具体可执行的工具和步骤

---

**总结对比：**
* 对比要点
```

### 内容质量规则

- 每个方法论章节必须包含 `**调研方法：**` 行，提供具体、可执行的工具或步骤——不要让方法停留在抽象层面
- 引用框架（PEST、Kano等）时，在上下文中简要解释；不要假设读者了解每个框架
- 尽可能包含实际案例（如"如电商APP加直播"）
- 尽可能量化（如"流失率高达70%"而非"流失率较高"）

## 工作流背景

本仓库记录了顽鹿的产品经理工作流，分为两种模式：

### 当前工作流程
串行评审关卡流程：
1. 市场调研 → 需求收集 → 概念设计
2. 概念评审（BU总经理 + 各中心负责人决策通过/驳回）
3. 原型设计 → 可行性评估 → 原型评审
4. 需求细化 → 视觉交互设计 → PRD评审
5. 需求宣贯 → 版本排期 → 发布上线

### 目标工作流（改造后的工作流）
开发中——目标是在上述每个阶段整合AI辅助能力。这是本仓库的核心目的。

## 如何使用本仓库

### 添加新内容

1. 创建一个新的 `.md` 文件，使用描述性的中文文件名
2. 遵循上述文档结构模式
3. 如果文档覆盖主要工作流阶段，在 `README.md` 中添加链接
4. 每个文件专注于一个方法论或工作流阶段

### 编辑现有内容

- 保留 README 中现有的层级符号风格，方法论文档中使用 `*`
- 不要大规模重新格式化现有文档——匹配所编辑文件的本地约定
- 在 `README.md` 工作流章节添加内容时，保持编号步骤格式

### 禁止事项

- 未经明确批准，不要向本仓库添加代码、脚本或自动化
- 不要将中文内容转换为英文
- 不要重构 `README.md` 工作流描述的层级结构
- 不要删除方法论文档中的 `调研方法` 行
- 未经确认存储策略前，不要添加图片或二进制文件
